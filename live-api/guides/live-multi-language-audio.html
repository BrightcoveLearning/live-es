    Audio multilingüe en vivo

    En este tema se explica cómo crear un trabajo en vivo con varias pistas de audio en diferentes idiomas.

    Requisitos

    • Una cuenta configurada para la Live API con función de audio multiidioma habilitada (póngase en contacto con su administrador de cuentas para obtener más información)
    • Una clave de API para la API en vivo (que se le proporciona cuando se configura la cuenta)
    • Un codificador que admite el protocolo RTP. El soporte de audio multilingüe está limitado a rtp o rtp-fec protocolo.

    Crea tu trabajo en vivo

    Para agregar varias pistas de audio a su trabajo, se debe proporcionar una lista de pistas de audio en el cuerpo de la solicitud al crear un trabajo en vivo:

    {
  "live_stream": true,
  "region": "us-west-2",
  "protocol": "rtp",
  "cidr_whitelist": [ /* omitted… */ ],
  "outputs": [ /* omitted… */ ],
  "alternate_audio": {
    "tracks": [ { Track Details } ]
  }
}

    Campos admitidos para el Track objeto

    La siguiente tabla contiene una descripción completa de los campos admitidos en el track objeto

    Campos de seguimiento
    Campo Descripción
    pid Requerido. El PID para seleccionar del TS entrante para la pista que se especifica en el archivo de flujo enviado a Wowza.
    language Requerido. El código para el idioma que se va a utilizar. En la actualidad, esto es flexible y puede estar en cualquiera de los diferentes formatos de idioma requeridos/soportados por las especificaciones, específicamente RFC5456/BCP47 que también cubre ISO-639 según los requisitos. https://tools.ietf.org/html/rfc5646

    > LANGUAGE Campo HLS

    > AdaptationSet:lang Campo DASH
    variant Opcional. Asigna a los valores del esquema de rol DASH: main, alternativo, comentario, suplementario, dub. Se utiliza en el rol DASH urn:mpeg:dash:role:2011. El valor predeterminado es main.

    HLS > CARACTERÍSTICAS ver https://tools.ietf.org/html/rfc8216#section-4.3.4.1
    label Opcional. Una descripción de texto que se utilizará para la pista, preferido ser utilizada por el jugador siempre que sea posible. Utilizado en: > NAME Campo HLS. El valor predeterminado es language con un ID para cada pista, por ejemplo, en-0 si no se especifica.
    streams [] Opcional. Si no está presente copia, utilice sólo los ajustes de audio predeterminados.

    HLS > GROUP-ID. (Objeto)

    Ejemplo: { “source”: “720p” } - coincidir la etiqueta de la fuente de salida.

    Esto imita el streams campo del tipo de salida de la lista de reproducción.
    default Necesario excepto donde es predeterminado. Qué pista de audio debe marcarse como DEFAULT en el, así EXT-X-MEDIA como en qué muxed en secuencia de vídeo*

    Si no está presente, primero es el predeterminado.

    * si se definen varias listas de reproducción con diferentes valores predeterminados, la pista predeterminada de nivel superior es la muxed

    Configurar el codificador

    Configure su codificador para entregar MPEG-TS a través de RTP a stream_url desde la respuesta de creación. Asegúrese de que las pistas de audio proporcionadas en el cuerpo de la solicitud estén configuradas con el PID correcto en el codificador.

    Probar la reproducción

    Cargue la playback_url en Brightcove Player o Safari para confirmar la reproducción.

    Reproducción

    Actualmente, DRM solo se admite a través del servicio Catch-up (Timeshift). Agregue estos campos a su cuerpo de solicitud de solicitud de creación de trabajo:

    {
  "drm": {
    "modes": [
      "all"
    ]
  },
  "timeshift": {
    "type": "single",
    "startover_duration": 36000,
    "skip_live_output": true
  }
}

    La respuesta incluirá salidas DRM y una timeshift_url para usar para la reproducción.
