    Resumen: API en vivo de Brightcove

    En esta descripción general, aprenderá para qué sirve la API Live y cómo usarla. Los temas incluidos en este documento incluyen regiones y CDN de AWS compatibles, canales en vivo y eventos, así como la inserción de metadatos temporizados ID3 en una transmisión en directo.

    Introducción

    El Live API es una API basada en REST que le permite crear y administrar eventos de transmisión en vivo. Entre las características opcionales se incluyen:

    • Inserción de anuncios en el lado del servidor ( SSAI )
    • AES-128 encriptación
    • Crear activos de vídeo bajo demanda a partir de clips tomados de la transmisión en directo
    • DVR capacidad
    • Múltiples CDNs

    Consulte también la Referencia de API.

    URL base

    La URL base para el Live API es:

          https://api.bcovlive.io

    Encabezados

    Todas las solicitudes se autentican mediante una clave de API que se le proporcionará cuando se configure su cuenta. La clave se pasa en un X-API-KEY encabezado. También se requiere un Content-Type encabezado:

          X-API-KEY : YOUR_APIKey
      Content-Type: application/json

    Regiones de AWS admitidas

    Se admiten las siguientes regiones de AWS:

    Regiones de AWS admitidas
    Ubicación Nombre de AWS Soporte para SSAI
    Oregon us-west-2
    Virginia us-east-1
    Tokio ap-northeast-1
    Singapur ap-southeast-1
    Sídney ap-southeast-2
    Bombay ap-south-1
    Frankfurt eu-central-1
    Irlanda eu-west-1

    Tenga en cuenta que los trabajos de SEP están limitados por cuenta donde el límite estándar es 3 excepto para el us-west-2 que su limitación es de hasta 10. Todas las limitaciones se establecen por cuenta y no por región.

    Redes de distribución de contenidos (CDN) compatibles

    Las siguientes CDN son compatibles con la transmisión en vivo:

    • Akamai
    • Cloudfront

    Otras CDN basadas en archivos deberían funcionar, pero no se han probado y no se admiten activamente.

    Canales y horas de eventos

    Hay dos opciones de compra para Live:

    • Horas de eventos de compra de tiempo de streaming
    • Comprar canales de streaming

    También puede comprar tanto las horas de transmisión de eventos como los canales. Póngase en contacto con su administrador de cuenta para obtener más información sobre las ofertas.

    Si ha adquirido tanto las horas de transmisión de eventos como los canales, debe tener en cuenta lo siguiente:
    • De forma predeterminada, los trabajos en vivo se facturan en comparación con las horas de transmisión de eventos.
    • Si desea que un trabajo se facture como trabajo de canal, debe incluir el siguiente campo en el cuerpo de la solicitud al crear el trabajo activo:

      "channel_type": "24x7"

    • Los clientes de solo canal deben incluir siempre el siguiente campo en el cuerpo de la solicitud al crear el trabajo activo:

      "channel_type": "24x7"

    • Los clientes que hayan comprado sólo horas de eventos de tiempo de streaming, no deben incluirlos "channel_type": "24x7" en las solicitudes.

    autenticación de tokens

    Brightcove ofrece la opción de agregar autenticación de token a las URL de reproducción de secuencias de vídeo en vivo. Si desea agregar autenticación de token, póngase en contacto con el soporte técnico de Brightcove. La autenticación de tokens puede tardar hasta tres días en configurarse.

    El TTL (tiempo de vida) de los tokens se puede establecer en cualquier valor de una hora a 365 días. El tiempo que establezca el TTL dependerá de los tipos de transmisiones en vivo que implemente. Tenga en cuenta que el TTL es una configuración de toda la cuenta, sin embargo, y se aplicará a todas las transmisiones en vivo.

    DVR capacidad

    Las transmisiones en vivo de Brightcove tienen DVR capacidad. Para utilizar esta capacidad, debe:

    • Usar la playback_url_dvr URL para la reproducción
    • Utilizar un reproductor que tenga capacidad de DVR

    La capacidad de DVR está limitada a 86.400 segundos.

    La DVR transmisión permanecerá disponible durante 7 días después de que se complete la transmisión en vivo.

    Endpoints y operaciones

    Las operaciones principales para el Live API son crear y administrar transmisiones en vivo, y generar clips VOD a partir de transmisiones en vivo. Estas operaciones se llevan a cabo a través de solicitudes a los siguientes puntos finales, que se explican con más detalle en el resto del documento.

    Tenga en cuenta que Brightcove Live tiene su propio ID de cuenta, independiente de cualquier otro ID de cuenta. En las operaciones de API, account_id hace referencia a su ID de cuenta Live.

    Creación y gestión de puestos de trabajo

    Estas operaciones le permiten crear un trabajo en vivo, obtener los detalles del mismo y detenerlo. También hay un punto final para crear un punto de referencia inmediato para un salto de anuncio.

    Crear un trabajo activo

          POST https://api.bcovlive.io/v1/jobs

    Este punto final se utiliza para crear transmisiones en vivo a través de una POST solicitud. Además de especificar las propiedades de la transmisión en vivo en sí, la solicitud también puede especificar clips VOD que se generarán a partir de la transmisión en vivo (esto también se puede hacer más adelante a través del punto final). Los detalles de los campos que se pueden incluir en el cuerpo de la solicitud se proporcionan en la Referencia de API.

    Protocolo de entrada

    Brightcove Live admite múltiples protocolos de entrada. Utilice el protocol campo en el cuerpo de la solicitud cuando cree el trabajo para especificar el que va a utilizar. Los valores admitidos son:

    • rtmp ( el valor predeterminado)
    • rtp
    • rtp-fec
    • srt

    El protocolo RTMP es para la entrega de una secuencia en formato FLV. Los otros protocolos son para entregar MPEG2-TS.

    Si utiliza rtp, rtp-fec o srt, también debe especificar un cidr_whitelist(consulte Enrutamiento entre dominios sin clase).

    Si utiliza rtmp, puede especificar un ip_whitelist para la entrada en su lugar, pero esto no es necesario.

    Ejemplo de cuerpo de solicitud para el trabajo RTP+FEC:

    {
    "live_stream":true,
    "region":"us-west-2",
    "reconnect_time":300,
    "outputs":[
      {
        "label": "hls720p",
        "live_stream": true,
        "height": 720,
        "video_bitrate": 800,
        "segment_seconds": 6,
        "keyframe_interval": 90
      }
    ],
    "protocol": "rtp-fec",
    "cidr_whitelist": ["127.0.0.1/32"]
}

    El inicio Live API rápido le guiará a través de la creación de un trabajo de transmisión en vivo y la configuración de un reproductor de Brightcove para reproducirlo.

    Listar trabajos en vivo

          GET https://api.bcovlive.io/v1/jobs

    Este punto final se utiliza para listar sus transmisiones en vivo a través de una GET solicitud. El endpoint admite la paginación, la ordenación y el filtrado de búsquedas. Los detalles de los campos que se pueden incluir en el cuerpo de la solicitud se proporcionan en la Referencia de API y se puede encontrar información adicional en Obtener una lista de trabajos en vivo o VOD.

    Obtener detalles de trabajo en vivo

          GET https://api.bcovlive.io/v1/jobs/:jobId

    Este extremo le permite obtener la información detallada sobre una transmisión en vivo, que también se devuelve cuando se crea originalmente el trabajo. Consulte la Referencia de API para obtener detalles sobre los campos de respuesta.

    Inserción manual de puntos de referencia de anuncio

          POST https://api.bcovlive.io/v1/jobs/:jobId/cuepoint

    Normalmente, el codificador enviará puntos de referencia para los saltos de anuncios, pero también puede crear un salto de anuncio inmediato enviando una solicitud a este extremo. Consulte la Referencia de API para obtener más detalles.

    Tenga timecode en cuenta que DD:HH:MM:SS se requiere un en el formulario para el punto de referencia.

    Detener un trabajo en vivo

          PUT https://api.bcovlive.io/v1/jobs/:jobId/cancel

    Utilice este extremo para detener una transmisión en directo inmediatamente. Una vez cancelada, no se puede reiniciar una transmisión en directo. Consulte la Referencia de API para obtener más detalles.

    Creación de clips

    Puede crear clips de vídeo bajo demanda desde una transmisión en directo y almacenarlos en una cuenta de Video Cloud, o enviarlos a un bucket de S3 o a una dirección FTP. Puede definir los clips al crear la transmisión en directo o crearlos más tarde utilizando el punto final descrito a continuación. Consulte también la guía Creación de clips .

    Crear clip VOD

          POST https://api.bcovlive.io/v1/vods

    Los puntos inicial y final de los clips se pueden definir en términos de desplazamientos desde el inicio de la secuencia o marcas de tiempo UNIX. Los detalles de los campos del cuerpo de la solicitud se pueden encontrar en la Referencia de API.

    Tenga en cuenta que los clips de transmisiones en directo se pueden crear hasta 7 días después de que finalice el evento.

    Obtener una lista de trabajos VOD (clip)

    Para obtener una lista de los trabajos VOD para clips, consulte Obtención de una lista de trabajos en vivo o VOD y Referencia de API.

    Administrar SSAI

    Mediante la inserción de anuncios del lado del servidor ( SSAI), puede insertar tantos saltos de anuncios como desee en la transmisión en directo. También puedes ingerir recursos de pizarra (clips VOD) para rellenar cualquier tiempo de anuncio no utilizado con un mensaje a la derecha o lo que quieras.

    SSAI Encontrará más detalles sobre la configuración en Inserción de anuncios del lado del servidor usando Brightcove Live API y la Referencia de API.

    Obtener configuraciones de anuncios de cuenta

          GET https://api.bcovlive.io/v1/ssai/applications/:account_id

    Este endpoint le permite obtener todas las configuraciones de anuncios que se han configurado para una cuenta. Los detalles de los campos de respuesta se pueden encontrar en la API Reference.

    Crear configuración de publicidad

          POST https://api.bcovlive.io/v1/ssai/application

    Cree una configuración de anuncio que defina cómo se recuperarán los anuncios SSAI. Los detalles de los campos del cuerpo de la solicitud se pueden encontrar en la Referencia de API.

    Obtener configuración de anuncios

          GET https://api.bcovlive.io/v1/ssai/application/:application_id

    Utilice este endpoint para obtener los detalles de una configuración de anuncio que ha creado. Los detalles de los campos de respuesta se pueden encontrar en la API Reference.

    Actualizar configuración de anuncio

          PUT https://api.bcovlive.io/v1/ssai/application/account/:application_id

    Actualizar los detalles de una configuración de anuncio. Los detalles de los campos del cuerpo de la solicitud se pueden encontrar en la Referencia de API.

    Obtener activos de origen de medios de pizarra

          GET https://api.bcovlive.io/v1/ssai/slates/:ACCOUNT_ID

    Obtenga los activos de medios de pizarra que se han definido para una cuenta. Los recursos multimedia de pizarra se utilizan para rellenar el tiempo de descanso de anuncios que no se rellenan con los anuncios. Los detalles de los campos de respuesta se pueden encontrar en la API Reference.

    Ingesta de recursos de origen de medios de pizarra

          POST https://api.bcovlive.io/v1/ssai/slates

    Añada un recurso multimedia para las pizarras para rellenar el tiempo de descanso de anuncios sin rellenar. Los detalles de los campos del cuerpo de la solicitud se pueden encontrar en la Referencia de API.

    Eliminar recurso de origen de medios de pizarra

          DELETE https://api.bcovlive.io/v1/ssai/slates/:SLATE_MSA_ID

    Elimina un recurso multimedia de pizarra.

    Puntos de entrada estáticos

    La función de puntos de entrada estáticos (SEP) permite un trabajo en vivo de larga duración que se puede activar y desactivar mientras mantiene las URL de punto de entrada y las URL de reproducción estáticas y reutilizables. Esta función permite a los clientes configurar su codificador en sus instalaciones o en el campo y permite al cliente crear su propia lógica de programación para canales o programas en vivo. Consulte Puntos de entrada estáticos para obtener más detalles.

    Subtítulos

    Si los subtítulos están dentro de la señal de entrada h264 (señalizada correctamente en el paquete user_data), éstos se pasan a través de las salidas h264.

    Si está utilizando un codificador en vivo de emisión Elemental, puede obtener subtítulos de SDI (EIA-608/CEA-608) u otras fuentes (SCTE-20, SCC, Teletexto, DVB-sub, Ancillary, ARIB, TTML, SCTE-27, STL, SRT, SMI) y ponerlos en el flujo h264 que nos envía. Otros codificadores de grado de difusión probablemente puedan hacer lo mismo, pero no los hemos probado formalmente.

    Esta información se ha movido a Insertar metadatos temporizados ID3.

    Limitaciones

    • Para que los trabajos Live creados con la API aparezcan y no se puedan utilizar en el Módulo Live, debe incluir el videocloud objeto en el cuerpo de la solicitud al crear el trabajo.

      Por ejemplo:

      {
  "live_stream": true,
  "region": "eu-central-1",
  "reconnect_time": 1800,
  "live_sliding_window_duration_ms": 0,
  "outputs": [
    { "label": "hls720p", "live_stream": true, "width": 1280, "height": 720, "video_codec": "h264", "h264_profile": "high", "video_bitrate": 2100, "segment_seconds": 4, "keyframe_interval": 60 }
    ,
    { "label": "hls540p", "live_stream": true, "width": 960, "height": 540, "video_codec": "h264", "h264_profile": "main", "video_bitrate": 1500, "segment_seconds": 4, "keyframe_interval": 60 }
    ,
    { "label": "hls360p", "live_stream": true, "width": 640, "height": 360, "video_codec": "h264", "h264_profile": "main", "video_bitrate": 800, "segment_seconds": 4, "keyframe_interval": 60 }
  ],
  "videocloud": {
    "video":
      { "name": "live event UI", "description": "live event UI", "long_description": "", "tags": [], "reference_id": "", "state": "ACTIVE" }
    }
  }
    • La conexión inicial desde el codificador proporciona la información de ancho de banda que se va a crear con la lista de reproducción en vivo. Si la conexión inicial es baja, incluso si la configuración del trabajo tiene un alto rendimiento, la lista de reproducción mantendrá la misma información en la lista de reproducción hasta que se realice lo siguiente:
      • Se reinicia el codificador
      • Es posible que también deba borrarse la caché de CDN
    • Actualmente, la velocidad de fotogramas para las secuencias de entrada está limitada a 30 FPS. Si está interesado en utilizar una velocidad de fotogramas más alta, póngase en contacto con el Soporte técnico.
    • Por defecto, la resolución de la secuencia de entrada está limitada a 1080p.
    • No se deben modificar los ajustes de retransmisión durante el proceso de desconexión y reconexión. Cualquier cambio en el número de canales de audio, resoluciones o ajustes de códec provocaría un comportamiento imprevisible.
    • Aunque puede agregar DASH y MP4 para fuentes remotas para vídeos de Video Cloud, Live HLS solo admite actualmente.
    • Solo se admite audio AAC para las transmisiones de entrada.
    • Se permite un máximo de 5 trabajos activos en espera y sin iniciar en cualquier momento.

      Limitaciones adicionales en trabajos simultáneos:

      • El número de trabajos (24x7) está limitado a 0 o a un número bajo por región (dependiendo del tipo de cuenta).channel
      • El número de trabajos en ejecución simultánea está limitado por región, generalmente a 100.event
      • El número de trabajos en espera simultánea para conectar está limitado a 5.event
      • El número de trabajos de SEP por región está limitado a 3 ó 10 (consulte Regiones de AWS compatibles).

      Cualquiera de estos límites puede ser ajustado a nivel de cuenta por Soporte. Póngase en contacto con su administrador de cuenta si necesita capacidad adicional.

    • La dirección «RTMP» devuelta como stream_url para trabajos en vivo es una transmisión en vivo de Akamai HD, no una transmisión de FMS RTMP heredada; no es compatible con versiones anteriores de Internet Explorer.
    • Las transmisiones en directo se entregan a través de HTTPS y, si su cuenta de Brightcove no está habilitada para HTTPS, el reproductor de Brightcove no podrá cargar la transmisión en directo. Si su cuenta no tiene habilitada la compatibilidad con HTTPS para la publicación de origen, póngase en contacto con el servicio de soporte técnico de Brightcove para obtener la compatibilidad con HTTPS para la publicación de origen habilitada para evitar problemas de reproducción.
    • Cuando se utiliza una representación transmuxada dentro de una salida HLS de varias velocidades de bits, se segment_size puede incluir al transmuxar, pero debe establecerse de manera que sea un múltiplo del GOP tamaño de la secuencia de entrada. Por lo tanto, si la entrada es de 30 fps con fotogramas clave cada 60 fotogramas, el GOP tamaño es de 2 segundos, y el tamaño del segmento debe ser un múltiplo de 2. Si no lo hace, los segmentos de flujo serán de diferentes tamaños.

      También, keyframe_interval debería no ser especificado en cualquier salida.

    • Cuando utilice su propia ubicación de origen FTP o S3, su CDN debe configurarse para que se vuelva a su ubicación de origen. El sistema Brightcove Live no validará las ubicaciones de origen de las CDN proporcionadas en la solicitud de trabajo.
